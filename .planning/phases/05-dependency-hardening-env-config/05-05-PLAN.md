<!-- Per-batch protocol (D-01) -->

For each batch:
1. `pnpm update --latest <dep1> <dep2> ...` (or hand-edit package.json ranges + `pnpm install`)
2. `pnpm dedupe` (DEP-04)
3. `pnpm install --frozen-lockfile` (verifies the dedupe didn't leave drift)
4. `pnpm exec tsc --noEmit`
5. `pnpm lint`
6. `pnpm build` (D-05 floor)
7. `pnpm audit --prod --json | jq '.metadata.vulnerabilities'` (or non-JSON `pnpm audit`)
8. **Batch-specific smoke test** (see per-task spec)
9. Commit atomic: `chore(deps): bump <batch label> — closes Dependabot #X, #Y, ...`

<!-- Rollback protocol (D-02) -->

If any step fails: `git reset --hard HEAD~1` (if the batch was committed) OR discard the working tree changes (if not yet committed). No fix-forward in main.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| npm registry → repo (pnpm install) | Untrusted publisher network. Each `pnpm install` re-fetches metadata + tarballs. pnpm-lock.yaml integrity hashes (`integrity: sha512-...`) are the trust anchor. |
| Dependabot alert closer (`pnpm audit`) → developer's mental model | If `pnpm audit` says "0 vulnerabilities" but the lockfile was force-resolved through a merge conflict, the developer may believe the issue is closed when it is not (Pitfall 22). |
| Batch C package.json range edit → future Dependabot proposals | The tilde range pin `~12.23.26` plus the dependabot.yml ignore (Plan 01) together prevent silent regressions. Either alone is insufficient. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-14 | Tampering | A bumped transitive contains a malicious version (supply-chain attack) | mitigate | pnpm-lock.yaml integrity hashes verify every tarball against the registry hash at install time. `pnpm audit` after each batch surfaces known CVEs. No hand-editing of the lockfile beyond what pnpm produces. |
| T-05-15 | Denial of Service (UX) | Batch C framer-motion patch breaks the intro state machine despite passing tsc + lint (Pitfall 1) | mitigate | Mandatory three-state smoke test (Task 5 PART B) BEFORE merge: cold no-preference, cold reduced-motion, nav-back-from-/blog. Visual + behavioural confirmation; no automation can substitute (D-01). |
| T-05-16 | Elevation of Privilege | Batch B's `@auth/mongodb-adapter` bump (if bundled with mongodb) breaks the JWT callback's `isAdmin` assertion (Pitfall 2 cousin) | mitigate | If `@auth/mongodb-adapter` version moves in Batch B, AUTH-SMOKE-TEST.md MUST be re-run before merge (D-09). `next-auth` itself is not touched (Plan 01 ignore rule). |
| T-05-17 | Information Disclosure | Lockfile diff accidentally introduces a vulnerable transitive after a merge-conflict resolution (Pitfall 22 — silent vulnerability re-introduction) | mitigate | Strict protocol: never `rm pnpm-lock.yaml && pnpm install`; let pnpm reconcile via `pnpm install --frozen-lockfile=false` on resolution conflicts; `pnpm dedupe` after every batch; `pnpm audit` cross-checks the final state. |
| T-05-18 | Repudiation | Hard to trace which Dependabot PR closed which advisory after the dust settles | mitigate | Each batch commit message lists the closed Dependabot PR numbers in the trailer: `Closes #X, #Y, #Z`. Final SUMMARY.md tabulates batch → advisory IDs closed → CVE numbers. |
| T-05-19 | Denial of Service | A batch breaks Coolify production deploy after merge (e.g. Next 16.1.6 → 16.2.0 changes turbopack output) | mitigate | `pnpm build` is in the floor (D-05 — extends v1's tsc+lint). On Coolify redeploy failure, the rollback per D-02 is `git revert <merge commit>` and Coolify auto-redeploys the previous good build. |
| T-05-20 | Tampering | `rehype-sanitize` major bump silently changes the default schema (SEC-01 set an explicit allow-list — Batch D must verify that path still loads the explicit schema) | mitigate | Batch D smoke test (Task 6) includes a quick verification that `src/components/blog/MarkdownRenderer.tsx` still uses the explicit allow-list schema (SEC-01 from v1); if the import path or API changed, address inline. |
| T-05-21 | Information Disclosure | `pnpm audit` output includes registry URLs / package paths in stdout that leak into commit messages | accept | These are public registry URLs; no private metadata. Acceptable to include in commit body. |
</threat_model>
