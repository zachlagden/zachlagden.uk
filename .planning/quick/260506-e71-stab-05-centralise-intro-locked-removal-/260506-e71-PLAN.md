<objective>
Centralise the `intro-locked` body class lifecycle and the `#initial-loader` element so they exist ONLY on the home route. Today the root layout applies `intro-locked` and renders the loader on every route, forcing every non-home route (blog, admin, auth/signin) to render a `ClearIntro` helper or hand-rolled cleanup to undo it. After this change there is one owner of the home-route intro state and zero cleanup needed elsewhere.

Purpose: Eliminate the "remember to render `<ClearIntro/>` on every new top-level route" footgun documented in CONCERNS #14 and required by REQUIREMENTS STAB-05.

Output: One atomic commit moving the intro lock + loader off the root layout and onto the home route only, deleting `ClearIntro.tsx`, and removing all four call sites that previously cleaned up after the root layout.
</objective>

<approach_decision>
The task spec offered three approaches. The planner has locked in **approach (B): Bootstrap-only**.

Rationale (executor: do not re-evaluate, just implement):

1. **Approach (A) — Route group `(home)`** was the spec's "recommendation if all else equal", but it requires moving `page.tsx` and `HomeClient.tsx` into a new `src/app/(home)/` directory and adding a new `src/app/(home)/layout.tsx`. That is a file-move + new-layout-creation in a single atomic commit and broadens the blast radius. Imports via the `@/` alias would survive the move, but the diff becomes harder to review and there's a small risk of Next.js layout-nesting surprises (e.g. SessionProvider currently lives in root and a new home-only layout would need to NOT re-wrap it).

2. **Approach (C) — Pathname-aware bootstrap in root** keeps the loader shipping on every route (wasteful) and the body className becomes a client-side only mutation (FOUC risk on slow connections — body briefly has no `intro-locked` before the bootstrap mounts). Hard pass.

3. **Approach (B) — Bootstrap-only** is the minimal blast radius:
   - Root `layout.tsx` stops applying `intro-locked` and stops rendering the loader.
   - `page.tsx` (home route, server component) renders the `<div id="initial-loader">` directly into the SSR'd HTML — preserving pre-hydration loader coverage.
   - A new tiny client component `<HomeIntroBootstrap/>`, also rendered by `page.tsx`, uses `useLayoutEffect` to add `intro-locked` to `document.body` synchronously before first paint, and removes it on unmount (so client-side nav from `/` to `/blog` cleans up).
   - `Header.tsx`'s existing intro state machine (the canonical remover per commit `6edb503`) keeps removing the class when the intro finishes — its remove calls become idempotent no-ops if the bootstrap unmounts first, which is fine.

**Why useLayoutEffect, not useEffect**: `useLayoutEffect` runs synchronously after DOM mutations but before the browser paints. The body is already in the DOM (SSR'd without `intro-locked`); the bootstrap adds the class before the user sees anything, so there is no "unlocked then locked" flash. `useEffect` would run after paint and could allow a single frame of unlocked body during which scroll position could shift on slow devices.

**Why the loader stays in `page.tsx` server output, not in the bootstrap**: The whole point of the loader is to cover the screen *before* JS hydrates. If we rendered the loader from a client component, it wouldn't appear until after hydration — defeating the purpose. So `page.tsx` (RSC) emits the loader element directly, and the bootstrap only manages the body class lifecycle.

**FOUC analysis on body className**: Root layout no longer ships `intro-locked` on body. Between SSR paint and `useLayoutEffect` running, the body has no `overflow: hidden`. This is fine because:
   - Page content height under the loader doesn't matter — the loader's `position: fixed; inset: 0; z-index: 9999` covers everything.
   - The window can technically scroll for ~1 frame before hydration, but the user can't see anything other than the loader, and `Header.tsx` will adopt the `wasScrolledOnLoad` branch only if `window.scrollY > 0` at the moment its effect runs — which is identical to today's behaviour (today the user could also scroll a hidden body if JS hadn't run; `intro-locked` is a CSS class that only applies after CSS parses and body class is set, which on a normal cold load is essentially the same instant).
   - Net: no observable regression vs. current behaviour.

**Why this is one task, not two**: The work is mechanically a single semantic change — "move intro lock from global to home". Splitting (e.g. "first add bootstrap, then remove from layout") would create an intermediate state where intro-locked is applied twice. A single atomic commit is correct.
</approach_decision>

```tsx
"use client";

import { useLayoutEffect } from "react";

/**
 * Locks the body with `intro-locked` synchronously before paint on mount,
 * and removes the class on unmount (e.g. client-side nav away from home).
 *
 * The home route is the ONLY consumer of this component. Header.tsx's intro
 * state machine remains the canonical remover when the intro finishes; this
 * component handles the "applied before first paint" and "cleaned up on
 * route change" edges.
 */
export default function HomeIntroBootstrap(): null {
  useLayoutEffect(() => {
    document.body.classList.add("intro-locked");
    return () => {
      document.body.classList.remove("intro-locked");
    };
  }, []);

  return null;
}
```
